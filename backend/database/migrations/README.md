# Database Migrations

This directory contains database migration files for the Lomash Wood backend.

## Migration Files

Migrations are automatically generated by Prisma when you make changes to the schema:

```bash
# Generate migration after schema changes
pnpm db:migrate:dev

# Apply migrations to database
pnpm db:migrate

# Reset database and reapply migrations
pnpm db:migrate:reset
```

## Migration Naming

Migrations are automatically named with a timestamp and description:
- `20240101000000_init/` - Initial database schema
- `20240102120000_add_user_preferences/` - Added user preferences
- `20240103150000_update_product_schema/` - Updated product schema

## Running Migrations

### Development
```bash
# Create and apply migration
pnpm db:migrate:dev --name add_new_feature

# Apply pending migrations
pnpm db:migrate

# Reset database
pnpm db:migrate:reset
```

### Production
```bash
# Deploy migrations (requires DATABASE_URL)
pnpm db:migrate:deploy
```

## Seeding

After running migrations, seed the database with sample data:

```bash
# Run seed script
pnpm db:seed
```

## Migration Best Practices

1. **Always backup production database before migrations**
2. **Test migrations in staging first**
3. **Keep migrations reversible**
4. **Use descriptive migration names**
5. **Review generated migrations before applying**

## Troubleshooting

### Migration Conflicts
If multiple developers create migrations with the same timestamp:
1. Rename the migration directory
2. Update the migration file name
3. Re-run the migration

### Rollback Issues
If a migration fails:
1. Check the error message
2. Fix the issue in the migration
3. Run `pnpm db:migrate:reset` and reapply

### Schema Drift
If database schema doesn't match Prisma schema:
1. Run `pnpm db:push` to sync schema (development only)
2. Or create a migration to fix the drift
